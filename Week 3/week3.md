# Week 3 — Competency reflection (C2 & C3)

## C2: Code literacy and documentation

This week I strengthened my code literacy by carefully tracing bugs in a short analysis script until I understood what the code was doing and why the output was wrong. The first bug surfaced as a `ValueError` when the code tried to convert the string `"fifteen"` in `experience_years` to an integer. Working through that bug taught me to connect the traceback to a specific row in the CSV, recognize that the dataset mixed numeric strings with words, and decide how the program should behave when a value cannot be converted. The second error was difficult to find because the system did not recognize it as an error. The satisfaction scores were sorted in the default ascending order, so the script reported the five lowest scores while the comments and intent called for the five highest. Finding this error required reading the sort logic carefully rather than relying on an error message. After I understood each issue, I asked Cursor to help provide me with the correct syntax, fixed the code, and documented the fixed code with comments that describe stripping input, skipping invalid conversions, and sorting with `reverse=True` so the next reader (including me) can see how the script handles messy data and correct ordering.

## C3: Data cleaning and file handling

Debugging these two errors reinforced how data cleaning and correct file handling show up directly in analysis code, not only in a separate “cleaning” step. The `experience_years` problem is a cleaning issue that is very likely to appear in a real dataset. A column that should behave like numbers sometimes contains text. Finding these faulty data points meant attempting to convert each and every cell into integers, and skipping rows that could not be converted so that reading the CSV does not crash and averages are computed only from valid numeric values. The satisfaction ranking bug was about correctly interpreting sorted results once values were read from the file. The same CSV and the same integers can produce opposite conclusions if sort order is wrong. Together, those fixes remind me that loading a CSV is only the start. I need explicit rules for invalid values and for how rows are ordered before I attempt to analyze the data.
