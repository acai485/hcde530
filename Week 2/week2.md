# Week 2 — Competency 2: Code literacy and documentation

Observations for HCDE 530, based on Week 2 work (Python scripts, CSV data, and tooling).

---

## What “code literacy” meant for me this week

When I look at the Python files I have now, I can pick out the main syntax and follow how different parts of the code work. Cursor’s AI has been a big help for that: it lets me verify what I’m seeing and bridge gaps when something is unfamiliar. Over the course of the class session, that combination of reading actual code and getting targeted explanations made the files feel less opaque.

---

## What “documentation” meant to me this week

Documentation wasn’t only comments in the code. It also meant learning to write clear messages for each commit so future me (or collaborators) understands what changed and why. Helping set up the rules file was part of that too—capturing conventions and priorities so the project stays consistent.

---

## A concrete moment from my work

When I started commenting the Python file, I leaned on Cursor’s AI to explain what each chunk did. After doing that a few times, I began to recognize repeating patterns on my own and could write comments with less hand-holding. That shift—from “explain this block” to “I know what this block is doing”—felt like real progress on code literacy and documentation at the same time.

---

## Something that was hard or confusing

The hardest part was file paths and repo setup. I hadn’t used a repository before and was used to saving everything locally, so getting paths, folders, and version control aligned took extra effort and patience.

---

## What I want to keep doing next time

I want to keep naming and organizing files deliberately so work is easier to find and reuse later. Good structure pays off the first time I have to open an old script or share a folder with someone else.

---

## Connection to design / research

I can carry what I learned about reading code and about content engineering (clear structure, naming, and explanation) into vibe coding later on: I expect to get faster, more useful answers from AI tools when I can describe what the code is doing and what I want changed. That same literacy should make it easier to spot where the model and I disagree—to check its suggestions against the actual file and push back with specifics. For design and research, that means less time fighting the tool and more time using it to explore ideas, prototypes, and analysis with confidence.
